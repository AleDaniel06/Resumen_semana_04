Bloque 01 · Diseño

Diseño de Arquitectura
de Base de Datos

🏛️ 1.1 Tipos de Arquitecturas
Las más representativas
C
Centralizada
Toda la información en un único servidor central. Garantiza control centralizado pero presenta problemas de escalabilidad y riesgo de falla única. Ej.: sistema académico con servidor interno institucional.
CS
Cliente-Servidor
El servidor aloja la BD y procesa solicitudes; los clientes generan consultas y reciben resultados. Mejora la distribución de funciones. Ej.: sistema de ventas donde los puntos de caja consultan inventarios en tiempo real.
D
Distribuida
Datos fragmentados/replicados en distintos servidores geográficos. Mejora el rendimiento y disponibilidad. Ej.: bancos multinacionales con sucursales y acceso global.
☁
En la Nube
Servicios gestionados por proveedores cloud (AWS, Azure, Google Cloud). Ofrece escalabilidad dinámica y reducción de costos. Ej.: e-commerce usando Amazon RDS.
⚖️ 1.2 Criterios de Selección
Factores clave para elegir
✦ Escalabilidad y rendimiento: arquitecturas distribuidas o nube para gran volumen y concurrencia
✦ Disponibilidad y tolerancia a fallos: entornos críticos requieren redundancia y replicación
✦ Seguridad y normativas: almacenamiento centralizado facilita control; cloud debe cumplir GDPR
✦ Costo y recursos: distribuidas implican mayor inversión; nube ofrece costos variables bajo demanda
✦ Contexto organizacional: pequeña empresa → cliente-servidor local; corporación global → nube distribuida
📐 1.3 Modelado Conceptual y Lógico
Dos niveles esenciales
🔵 Modelado Conceptual
Describe la información a alto nivel usando diagramas entidad-relación (E-R) para representar entidades, atributos y relaciones, independientemente del DBMS. Ej.: en una biblioteca → Libro, Usuario, Préstamo.
🟡 Modelado Lógico
Traduce el modelo conceptual a estructuras compatibles con un DBMS. Define tablas, claves primarias, claves foráneas e integridad referencial. Ej.: entidad Cliente → tabla con ID, nombre y correo, relacionada con Pedido.
🔧 1.4 Normalización y Diseño Físico
Optimización del diseño

La normalización organiza datos siguiendo formas normales (1FN, 2FN, 3FN) para eliminar redundancia y asegurar integridad. El diseño físico traduce el modelo lógico en estructuras específicas del DBMS: tablas, índices, particiones, estrategias de almacenamiento.

La optimización del diseño busca mejorar el rendimiento: uso adecuado de índices, particionamiento y replicación en arquitecturas distribuidas, y balance entre normalización y desnormalización según la carga del sistema.
03
Bloque 02 · Implementación

Implementación en Entorno
de Desarrollo

La implementación es la fase crítica donde se materializan las decisiones de diseño lógico y físico, asegurando que los datos estén organizados de manera eficiente, segura y escalable.

⚙️ 1.1 Instalación y Configuración
Primer paso de la implementación

Dependiendo del proyecto se puede optar por MySQL, PostgreSQL, SQL Server, Oracle o MongoDB. La configuración inicial incluye:

✦ Definir usuarios y roles con privilegios adecuados
✦ Configurar parámetros de seguridad y rendimiento (memoria asignada, número de conexiones, logs de auditoría)
✦ Crear una base de datos vacía como entorno de trabajo
Ej.: estudiantes instalan PostgreSQL localmente y configuran usuario con privilegios restringidos para garantizar buenas prácticas de seguridad.
🗂️ 1.2 Creación de Estructuras
Almacenamiento y organización

Una vez configurado el DBMS, se definen las estructuras que permitirán almacenar y organizar los datos:

📋 Tablas
Representan entidades del modelo lógico. Ej.: Libros, Usuarios, Préstamos en una biblioteca.
👁️ Vistas
Consultas guardadas que presentan información derivada. Ej.: "Libros disponibles".
⚡ Índices
Estructuras auxiliares que optimizan la velocidad de consulta. Ej.: índice sobre ISBN para agilizar búsquedas.
🔗 1.3 Relaciones e Integridad
Restricciones de integridad

Las restricciones aseguran que los datos almacenados mantengan consistencia y correspondan con las reglas de negocio:

🔑 Clave primaria (PK)
Identifica de manera única cada registro en la tabla.
🔗 Clave foránea (FK)
Establece relaciones entre tablas. Ej.: Préstamos tiene FK hacia Usuarios y FK hacia Libros.
✅ Restricciones adicionales
Unicidad, no nulos y validación de dominio para garantizar calidad del dato.
📥 1.4 Carga Inicial de Datos y Pruebas Preliminares

La carga de datos inicial permite poblar las tablas con información representativa, lo que facilita la validación temprana de la base de datos. Se ejecutan pruebas de: integridad referencial (comprobar que las FK funcionen), consultas de validación (verificar tiempos de respuesta y resultados correctos) y pruebas de inserción, actualización y eliminación.

Ej.: estudiantes insertan 100 registros de libros, 50 usuarios y 200 préstamos, luego ejecutan consultas como "préstamos vigentes" para validar la integridad y eficiencia del sistema.

Bloque 03 · Validación

Validación de la
Arquitectura Implementada

La validación garantiza que el sistema responda adecuadamente a las necesidades de los usuarios finales, manteniendo la integridad de los datos, optimizando el rendimiento y evitando errores que puedan comprometer la calidad del servicio.

🔍 1.1 Pruebas de Integridad de Datos
El pilar fundamental

La integridad consiste en asegurar que los datos cumplen con las restricciones de unicidad, obligatoriedad, dominio y consistencia referencial definidas en el modelo lógico.

✦ Verificar que no se inserten registros duplicados en clave primaria
✦ Comprobar que valor de clave foránea exista en la tabla referenciada
✦ Validar que no se ingresen valores nulos en atributos obligatorios
Ej.: en BD académica, validar que ningún estudiante pueda registrarse en matrícula sin existir previamente en la tabla de estudiantes y que no haya duplicidad en DNI.
⏱️ 1.2 Evaluación de Consultas y Tiempos de Respuesta
Medir la eficiencia

La eficiencia depende de la rapidez y exactitud con que la BD responde a las consultas. Durante la validación se realizan pruebas que incluyen:

✦ Medición de tiempos de ejecución de consultas simples y complejas
✦ Pruebas de carga con múltiples usuarios concurrentes
✦ Comparación de rendimiento antes y después de crear índices o ajustar consultas
Ej.: en BD de e-commerce, se evalúa cuánto tarda una consulta que lista los productos más vendidos en los últimos 6 meses. Se mide nuevamente tras implementar índices para comprobar la mejora.
🚦 1.3 Identificación de Cuellos de Botella y Optimización
Detectar y resolver limitaciones

Los cuellos de botella afectan el rendimiento general. Pueden encontrarse en: consultas mal diseñadas, índices mal implementados, configuración deficiente del servidor o excesiva carga de usuarios simultáneos.

La optimización implica técnicas como:

✦ Reescribir consultas SQL de forma más eficiente
✦ Implementar índices adicionales o eliminar los innecesarios
✦ Mejorar la normalización o desnormalización según el caso
✦ Ajustar parámetros del servidor DBMS para mayor rendimiento
Ej.: en un sistema hospitalario, una consulta de reportes de pacientes tarda demasiado por incluir subconsultas anidadas. Tras reescribirla con JOINs y optimización de índices, el tiempo se reduce en un 70%.

Bloque 04 · Evaluación

Evaluación de Funcionalidad
y Eficiencia

Una BD no solo debe almacenar información de manera coherente, sino también ofrecer rendimiento óptimo, confiabilidad y escalabilidad, de forma que las operaciones de lectura, escritura y actualización respondan adecuadamente a los requerimientos del negocio.

📊 1.1 Técnicas de Monitoreo y Métricas de Rendimiento
¿Cómo medir la eficiencia?

El monitoreo implica la observación constante del comportamiento del sistema en diferentes escenarios de carga. Métricas más relevantes:

⏱️
Tiempo de respuesta de consultas
🔄
Throughput (transacciones/seg)
💻
Uso de CPU y memoria
💾
Índices de E/S de disco
🔒
Bloqueos y concurrencia
📈
Rendimiento global del sistema
Ej.: un sistema hospitalario monitorea consultas a la tabla historial_clínico que son lentas debido a la ausencia de índices adecuados.
🔄 1.3 Retroalimentación y Mejoras Continuas
Ciclo PDCA aplicado a la BD

La retroalimentación es un proceso cíclico inspirado en metodologías ágiles y de mejora continua como el PDCA (Plan-Do-Check-Act). Ejemplos de acciones de mejora:

✦ Ajustar índices para consultas críticas
✦ Optimizar el diseño de tablas mediante normalización o desnormalización estratégica
✦ Redefinir políticas de backup y recuperación ante fallos
✦ Reestructurar consultas SQL para mejorar tiempos de ejecución
Ej.: en sistema de ventas en línea, tras recibir retroalimentación, los estudiantes crean índices compuestos en categoría y precio, mejorando el rendimiento en un 40%.
📄 1.2 Reporte de Evaluación Técnica
Documento formal de hallazgos

El reporte técnico registra los hallazgos obtenidos durante la validación del sistema. Un buen reporte debe incluir:

🖥️ Descripción del entorno de pruebas
Hardware, software y configuración del DBMS utilizado.
📏 Indicadores medidos
Tiempos promedio de respuesta, consumo de recursos, integridad de los datos.
📊 Resultados obtenidos
Tablas comparativas, gráficos y análisis descriptivo de los datos recogidos.
✅ Conclusiones y recomendaciones
Acciones a seguir para mejorar la eficiencia del sistema.
Ej.: estudiantes elaboran reporte técnico sobre rendimiento de PostgreSQL al ejecutar 10.000 transacciones, incluyendo gráficas comparativas de consultas con y sin índices.

Este proceso se integra en proyectos de desarrollo bajo la estrategia ABP (Aprendizaje Basado en Proyectos), permitiendo a los estudiantes aplicar los fundamentos teóricos en proyectos reales o simulados.

📐 Actividades · Diseño de Arquitectura
Selección y diseño de arquitecturas
1. Analizar qué arquitectura usar en un hospital regional
Se recomienda una arquitectura cliente-servidor con extensión distribuida. El servidor central gestiona los datos clínicos (historiales, diagnósticos, medicamentos). Múltiples clientes acceden desde consultorios, urgencias y laboratorios. Los módulos críticos (UCI, emergencias) se replican en nodos distribuidos para garantizar disponibilidad continua. Esta combinación responde a los criterios de: alta disponibilidad, seguridad (datos sensibles de pacientes), escalabilidad ante crecimiento de usuarios y tolerancia a fallos en servicios críticos.
2. Proponer criterios de selección para una startup de software educativo
Para una startup educativa se recomienda arquitectura en la nube (AWS/GCP). Criterios: (1) Escalabilidad dinámica: usuarios crecen rápidamente en períodos de exámenes; (2) Costo variable: pagar solo por recursos usados, ideal para startup sin gran capital inicial; (3) Disponibilidad global: estudiantes de distintos países acceden desde cualquier dispositivo; (4) Integración con servicios cloud: Amazon RDS o Google Cloud SQL facilitan la gestión sin personal de infraestructura dedicado.
3. Determinar la mejor arquitectura para empresa con sucursales internacionales
Se recomienda arquitectura distribuida con replicación. Justificación: (1) Los datos de cada sucursal se almacenan localmente para acceso rápido, y se replican al servidor central; (2) Si una sucursal pierde conexión, opera de forma autónoma; (3) Las consultas globales se distribuyen entre nodos para mejorar el rendimiento; (4) Cumple normativas locales de protección de datos (GDPR en Europa, por ejemplo). Tecnología recomendada: PostgreSQL con replicación streaming + MongoDB Atlas para datos no estructurados de interacciones.
⚙️ Actividades · Implementación
Instalación, estructuras y relaciones
1. Construir un diagrama E-R de un sistema de transporte urbano
Entidades: Ruta(id_ruta, nombre, origen, destino), Bus(id_bus, placa, capacidad), Conductor(id_conductor, nombre, licencia), Viaje(id_viaje, fecha, hora, ruta_fk, bus_fk, conductor_fk), Pasajero(id_pasajero, nombre, dni). Relaciones: Bus realiza Viaje (N:1), Conductor opera Viaje (N:1), Pasajero aborda Viaje (M:N → tabla ABORDAJE). Restricciones: PK en cada tabla, FK en Viaje hacia Ruta/Bus/Conductor, unicidad en placa del bus y DNI del conductor.
2. Transformar un modelo lógico en tablas físicas con SQL en un DBMS
SQL (PostgreSQL):
CREATE TABLE Conductor (id_conductor SERIAL PRIMARY KEY, nombre VARCHAR(100) NOT NULL, licencia VARCHAR(20) UNIQUE NOT NULL);

CREATE TABLE Viaje (id_viaje SERIAL PRIMARY KEY, fecha DATE NOT NULL, id_ruta INT REFERENCES Ruta(id_ruta), id_bus INT REFERENCES Bus(id_bus), id_conductor INT REFERENCES Conductor(id_conductor));

Se crean además índices: CREATE INDEX idx_viaje_fecha ON Viaje(fecha); para optimizar consultas por fecha de viaje.
3. Proponer un plan de índices y claves para optimizar una BD de e-commerce
Plan de optimización: (1) Índice compuesto en PEDIDO(id_cliente, fecha) para consultas de historial por cliente; (2) Índice en PRODUCTO(categoria, precio) para búsquedas filtradas; (3) FK entre PEDIDO y CLIENTE con ON DELETE RESTRICT para evitar pedidos huérfanos; (4) Índice de texto completo (GIN en PostgreSQL) en PRODUCTO(descripcion) para búsquedas por palabras clave; (5) Vista materializada VENTAS_DIARIAS actualizada diariamente para reportes de rendimiento sin recalcular joins.
🔍 Actividades · Validación
Pruebas e identificación de cuellos de botella
1. Medir el tiempo de respuesta de consultas simples y complejas en PostgreSQL
Metodología: Usar EXPLAIN ANALYZE en PostgreSQL. Consulta simple: SELECT * FROM Estudiante WHERE dni = '12345678'; → tiempo esperado <1ms con índice en dni. Consulta compleja: JOIN entre 4 tablas con GROUP BY y ORDER BY → medir antes (ej. 800ms) y después de crear índices compuestos (ej. 120ms). Registrar los planes de ejecución y comparar el número de filas analizadas (Rows) en ambos casos.
2. Detectar consultas lentas en un sistema académico y reescribirlas
Consulta ineficiente: SELECT * FROM Notas WHERE (SELECT promedio FROM Alumnos WHERE id = Notas.id_alumno) > 14; (subconsulta correlacionada → muy lenta). Consulta optimizada: SELECT n.* FROM Notas n JOIN Alumnos a ON n.id_alumno = a.id WHERE a.promedio > 14; Resultado: reducción del tiempo de 2.4s a 0.18s con índice en id_alumno. Mejora: 87% de reducción en tiempo de respuesta.
📊 Actividades · Evaluación y Mejora
Monitoreo, reporte y mejora continua
1. Comparar throughput de MySQL vs PostgreSQL usando los mismos datos
Escenario: 10.000 inserciones en tabla de pedidos con mismo hardware. Resultado esperado: PostgreSQL suele superar a MySQL en operaciones analíticas complejas (JOINs múltiples, agregaciones) y en cumplimiento de estándares SQL. MySQL puede ser más rápido en lecturas simples con MyISAM. Métricas a registrar: transacciones/segundo, tiempo total, uso de CPU (%), consumo de memoria (MB). Conclusión: PostgreSQL = mejor para BD transaccional académica; MySQL = mejor para aplicaciones web de alta lectura.
2. Implementar un ciclo PDCA en un proyecto de BD académica
Plan: Identificar que la consulta de "calificaciones por semestre" tarda 3.2 segundos. Do: Crear índice compuesto en (id_estudiante, semestre) y vista materializada CALIFICACIONES_SEMESTRE. Check: Medir nuevamente → 0.4 segundos. Mejora del 87.5%. Act: Documentar el cambio, aplicar mismo patrón a otras consultas frecuentes, redefinir política de actualización de vistas materializadas (diaria o en cada commit de notas). Repetir el ciclo en próxima iteración.
3. Rediseñar una consulta SQL ineficiente y demostrar la mejora obtenida
Consulta original (ineficiente):
SELECT * FROM Ventas WHERE YEAR(fecha_venta) = 2024 AND MONTH(fecha_venta) = 3;
(Función en columna → no usa índice).
Consulta optimizada:
SELECT * FROM Ventas WHERE fecha_venta BETWEEN '2024-03-01' AND '2024-03-31';
Resultado: de 1.8s a 0.09s con índice en fecha_venta. Mejora: 95%. Lección: evitar funciones sobre columnas indexadas en cláusulas WHERE.
